Error_Sender错误注入框架搭建
    为验证系统在持续异常数据条件下的鲁棒性，本阶段设计并实现独立错误注入工具：error_sender.py

    该工具参考第四阶段Sender → Communication Module验证架构进行扩展，但不模拟MCU，也不参与正常帧协议流程。

    系统运行过程：
        MCU仍按照真实Session周期运行；
        Communication Module 仍通过 UART 接收真实 JPEG 数据；
        AI推理流程保持不变；
        error_sender 仅通过独立 TCP 通道向 Communication Module 注入错误数据。
    
    Communication Module 内部保留 TCP Error Injection 接口：
        ENABLE_TCP_INJECT
    用于将错误数据混入现有接收处理流程。

    因此所有错误数据均经过：
        try_parse_frame()
        ↓
        图像分片重组
        ↓
        Session 状态机
        ↓
        超时清理
        ↓
        Session Abort
        ↓
        下一周期恢复逻辑
    从而能够观察系统面对连续异常数据时的真实行为。

Continuous Bad Frames 错误模型
    本阶段正式采用第四阶段提出的 Continuous Bad Frames（连续坏帧）抽象作为统一错误模型。

    其核心思想为，不穷举具体故障来源，而将：
        Camera 异常；
        JPEG 生成异常；
        UART 丢包；
        UART 抖动；
        分片损坏；
        数据缺失；
    等现实问题统一抽象为：
        “接收端持续获得一段不可用数据”。
    
    因此测试目标不是验证某一种具体错误，而是验证系统面对连续损坏数据时是否能保持：
        错误隔离；
        Session 边界完整；
        Buffer 清理；
        状态机恢复；
        下一周期重新同步。
    
    当前测试参数为：
        period = 17/13 s
        jitter = ±2 s
        burst = 8
        inter-frame-delay = 300 ms
        case = continuous_bad_frames
    
    通过非整数倍周期与随机抖动，使错误窗口逐步覆盖不同Session相位，而不仅仅局限于固定时间点。

错误恢复机制分析
    测试完成后，对日志与代码进行联合分析。

    确认系统主要依赖以下六层机制：
        1.底层帧解析和重同步机制
            第一层是 frame_codec.py (line 147) 里的 try_parse_frame()。
                它的作用是：坏数据进入 buf 后，不让解析器卡死，而是尽快丢弃不可用部分，继续寻找下一个 SOF = 0xAA55。
                日志中的这些错误都来自这一层：
                    PARSE_ERR: BAD_EOF_RESYNC
                    PARSE_ERR: BAD_VER_127_RESYNC
                    PARSE_ERR: BAD_LEN_2000001_RESYNC
                    PARSE_ERR: CRC_FAIL_DROP_FRAME
                    PARSE_ERR: NO_SOF_DROP_ALL
                代码对应关系：
                    NO_SOF_DROP_ALL：找不到 SOF，直接清空当前 buffer
                    BAD_VER_x_RESYNC：版本不对，丢 2 字节重新找 SOF
                    BAD_LEN_x_RESYNC：长度非法，丢 2 字节重新找 SOF
                    BAD_EOF_RESYNC：帧尾错误，丢 2 字节重新找 SOF
                    CRC_FAIL_DROP_FRAME：CRC 错，丢弃整帧
                所以parser层保证了：坏帧不会让接收端永久卡在某个错误 LEN、错误 EOF 或噪声块上。
        
        2. 图像分片重组的超时清理机制
            第二层是 image_chunk_protocol.py (line 408) 里的 ImageChunkAssembler.cleanup()。
            错误注入破坏部分分片后，某个 img_id 可能永远收不齐。此时 assembler 不会永久保留半张图，而是在超时后返回 expired id。
                主循环在pc_ai_uart_service_version2.py (line 914) 调用：
                    expired = assembler.cleanup()
                    if expired:
                        abort_current_session(...)
                日志中对应：
                    timeout drop img_id=[9]
                    SESSION ABORT: image assembly timeout img_id=[9]
                这层保证了：确实分片不会造成坏图像长期残留。
        
        3.上层分片语义异常触发 session abort
            第三层是主循环对 handle_image_frame() 的异常捕获，位置在 pc_ai_uart_service_version2.py (line 898) 附近。
            当注入的是“CRC 正确但语义错误”的 IMAGE_CHUNK，例如：
                chunk_seq=5, chunk_total=2
            底层帧解析会认为它是合法帧，但 parse_image_chunk_payload() 会判定语义非法。主循环捕获异常后直接 abort 当前 session。
            日志中对应：
                SESSION ABORT: image frame handling failed: ValueError: invalid chunk_seq/chunk_total: seq=5, total=2
            这层保证了：合法外壳但语义错误的分片不会继续污染当前图像重组状态
        
        4.session abort 会清空当前 PC 侧状态
            真正让系统“回得来”的核心，是 pc_ai_uart_service_version2.py (line 588) 的 abort_current_session()。
            它做了三件关键事：
                assembler.clear()
                last_pc_account = None
                ser.reset_input_buffer()
            也就是：
                清空图像分片缓存
                清空本轮 PC 对账状态
                清空串口输入缓冲中已经到达的残留字节
            日志中对应：
                SESSION ABORT: ...
                Drop current image data and return to STANDBY
            这层保证了：PC 不再试图修补当前坏 session，而是明确丢弃当前 session 的接收状态。
        
        5. 回到 STANDBY 的重新同步机制
            abort 后，主循环 continue，重新进入 wait_for_mcu_standby_and_send_ready() (line 649)。
            这个函数不要求前面的数据干净，只要在流中再次看到：
                STANDBY
            就发送：
                READY
            日志中非常典型：
                SESSION ABORT ...
                Waiting for MCU STANDBY...
                HANDSHAKE RX: b'... TX_DONE\r\n'
                HANDSHAKE RX: b'[TRANSFER_FAILED] ...'
                STANDBY received
                READY sent
                IMAGE COMPLETE: img_id=...
            这说明旧 session 的尾巴确实会进来，但最终靠下一轮 STANDBY 重新对齐。
            这层保证了：系统恢复的同步锚点不是坏帧后的某个字节，而是 MCU 下一轮 session 的 STANDBY 文本握手。
        
        6.TX_DONE 阶段的有限等待机制
            完成图像后，PC 进入 wait_for_transfer_done_and_send_ack() (line 690)。
            这里有两个恢复点：
                TRANSFER_DONE_TIMEOUT_SEC = 8.0
                如果看到 STANDBY，认为 TX_DONE 被下一轮替代，返回失败
            同时它会调用 service_result_requests() (line 448)，在等待 TX_DONE 的过程中继续处理 MCU 的 TYPE_REQ，所以 AI 结果轮询不会因为进入结束等待阶段而丢失。
            这层保证了：TX_DONE/DONE_ACK 阶段不会无限等，REQ/RESULT 也不会因为阶段切换被饿死。
        
    整体恢复链路：
        错误注入进入 buf
        -> try_parse_frame 重同步/丢坏帧
        -> 若分片缺失，assembler 超时
        -> 若分片语义错误，handle_image_frame 抛异常
        -> abort_current_session 清空 PC 当前 session 状态
        -> 主循环回到 Waiting for MCU STANDBY
        -> 旧 session 尾巴被读到但不形成新 session
        -> 下一轮 STANDBY 到达
        -> PC 发送 READY
        -> 新 session 正常 IMAGE COMPLETE
    
    当前错误恢复主要依赖三类机制：底层帧解析重同步、图像重组超时/异常清理、以及基于 STANDBY/READY 的 session 级重新同步。系统不是保证当前坏 session 一定成功，而是在判定当前 session 不可信后主动丢弃局部状态，并等待 MCU 下一轮 STANDBY 重新建立会话边界。因此在连续坏帧条件下，系统可以牺牲当前帧，但不会卡死、不会长期残留坏图像状态，也不会永久污染后续 session。

    AI 推理队列机制不属于错误帧恢复的直接机制。错误恢复主要由帧解析重同步、图像分片重组超时、session abort 清理和下一轮 STANDBY 握手完成。AI 队列的作用是将完整图片的推理过程从通信主循环中解耦，避免 AI 推理阻塞 UART 协议处理，从而间接提高系统在错误注入测试中的整体稳定性和恢复能力。

    系统鲁棒性并非来自一次性设计出的完整容错架构，而是由第四阶段形成的底层数据恢复链，与近期为解决 MCU-PC 通信时序问题而建立的会话级恢复机制共同叠加形成。
    这些最初用于解决具体“小问题”的机制，在完整系统中最终演化为一套跨越帧解析、图像重组、Session清理与通信重同步的多层恢复体系。

关于 IWDG 的观察
    本阶段原计划同时观察 IWDG 恢复行为。
    然而在累计数千个错误单元注入条件下，系统始终能够通过前置恢复机制完成恢复。

    因此：
        未观察到 MCU 死锁；
        未触发 IWDG 复位；
        未进入不可恢复状态。
    
    分析认为：
        IWDG 未被触发并非由于其失效，而是由于协议层恢复机制、Session Abort 机制以及 STANDBY 重同步机制已经能够在错误传播至死锁层之前完成恢复。
    
    因此当前系统鲁棒性主要依赖主动恢复机制实现，而 IWDG 作为最终硬件级保险保留。
