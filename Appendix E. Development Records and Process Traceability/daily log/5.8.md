日期：5.8

今日任务
    对此前基于Debug验证的摄像头测试工程进行重构，接触其与Debug模块的耦合，并尝试建立MCU→PC的真实图像UART上传链路

测试工程重构与Debug解耦
    首先，对此前用于摄像头单帧采集验证的测试工程进行了轻量化重构

    在此前工程中存在较强的Debug依赖关系，包括：
        camera_debug
        dbg_step
        dbg_*
        Debug 状态结构体等
    这些模块在摄像头Bring_up阶段用于观察VSYNC/HREF、PEG Marker、DCMI状态等问题具有重要作用；但在当前“真实通信链路验证”阶段，其职责已与主工程逻辑发生混杂，因此决定进行解耦。

    本次重构中：
        移除了camera_debug对主流程的依赖
        将采集状态结构体有Debug命名改为通用状态结构
        保留核心JPEG Buffer、Marker扫描与采集流程
        是main.c回归“调度器”角色
    
    重构完成后，摄像头采集流程重新回归：
        初始化 → 单帧采集 → JPEG Marker 扫描 → 获取 JPEG 长度
    的工程链路

PC通信模块重构
    随后，对此前阶段三/四中使用的PC通信模块进行了裁剪。

    由于当前目标仅为：
        验证 MCU 是否能够将真实摄像头图像上传至 PC
    因此暂时移除了：
        AI 推理
        REQ/RESULT 机制
        TCP 通道
        推理线程
        推理缓存机制
    保留： 
        frame_codec
        ImageChunkAssembler
        图像分片重组
        JPEG 文件保存
    因此形成了一个仅负责：
        UART接收 → 帧解析 → 图像重组 → JPG保存
    的轻量接收器

    此过程也进一步验证了此前阶段三中建立的：
        frame_codec.py
        image_chunk_protocol.py
    已经具备较好的协议层独立性，可脱离AI模块单独运行

MCU图像分片发送模块建立
    随后，在MCU端重新建立图像发送模块：
        uart_image_tx.c / .h
    并复用此前前阶段三/四建立的帧结构协议：
        SOF + VER + TYPE + SEQ + LEN + PAYLOAD + CRC32 + EOF
    同时，在 Payload 内部继续沿用 IMAGE_CHUNK 分片协议：
        IMG_ID + CHUNK_SEQ + CHUNK_TOTAL + CHUNK_DATA
    为保证 MCU 与 Python (zlib.crc32) 的 CRC32 结果一致，本阶段额外引入了 MCU 软件 CRC32 模块：
        crc32.c / .h
    并完成了：
        JPEG 分片
        CRC32 计算
        UART 阻塞发送
        Frame 打包
    等功能。

JPEG Buffer 与 Marker 验证
    在联调过程中，首先通过：
        Debug Watch
        Memory Browser
        STM32CubeProgrammer
    对 JPEG Buffer 状态进行了重新验证。

    验证结果表明：
        FF D8 FF E0
    JPEG 文件头能够稳定存在于 DCMI DMA Buffer 中，证明：
        OV2640 → DCMI → DMA → JPEG Buffer
    链路已稳定运行。

    随后发现：
        cam_state.jpg_len = 0
    导致UART不会真正发送图像数据

    经排查发现：
        CameraCapture_FindJPEGMarkers() 虽已完成 SOI/EOI Marker 扫描，但遗漏了 JPEG 长度计算逻辑。
    修复后，JPEG 长度恢复正常，系统可正确获得完整 JPEG 数据长度。

UART 图像上传链路验证成功
    在完成上述修复后，系统最终成功实现：
        OV2640
        → DCMI/DMA
        → JPEG Buffer
        → UART 分片发送
        → PC 帧解析
        → 图像分片重组
        → JPG 文件保存
    PC 端已能够成功接收到 MCU 上传的真实摄像头 JPEG 图像，并完成本地 JPG 文件保存。

    该结果意味着：
        真实图像链路已正式打通。

当前问题与阶段结论
    当前系统仍存在：
        发送端与接收端启动时序未同步
    的问题

    具体表现为：
        MCU 在程序启动后立即开始发送 JPEG
        PC 通信模块此时可能尚未完成串口初始化
        导致部分数据丢失，通信结果具有一定随机性
    经验证，该问题属于：
        通信触发机制问题
    而非：
        摄像头采集问题
        UART 硬件问题
        JPEG Buffer 问题
        帧协议问题
    
下阶段计划：
    引入 MCU ↔ PC 握手机制
    由 MCU 发送 "Stand by"
    PC 回复 "READY"
    MCU 收到 READY 后再开始发送 JPEG
    以解决当前启动时序错位问题。
