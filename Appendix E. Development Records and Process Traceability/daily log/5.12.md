系统架构调整：
    此前系统整体流程为：
        MCU 上电
        → 初始化摄像头
        → 等待 PC READY
        → 自动采集一张图像
        → 自动发送图像
        → 系统结束
    这样的结构能够验证：
        真实摄像头图像采集
        UART图像传输
        PC图像重组
    等核心链路
    但其本质仍属于“一次性线性流程”

    当前系统主循环结构调整为：
        while(1)
        {
            TriggerButton_Process();
            TriggerEvent_Process();
        }
    其中：
        TriggerButton：
            负责按键输入检测
            软件消抖
            状态切换
            事件触发

        TriggerEvent：
            负责拍摄事件执行
            图像发送事件执行
            图像缓存状态管理
    该结构标志着系统正式从：
        “启动即执行”
    转向：
        “输入事件驱动式系统”。

按键状态机建立：
    引入了上一阶段中已证明可行的状态机输入模块
    当前设计中：
        PH2：
            触发图像拍摄

        PH3：
            触发图像发送
    按键逻辑延续此前状态机测试工程中的方案：
        按键按下后不立即触发事件
    而是在
        检测按下
        → 软件消抖
        → 等待释放
        → 再次消抖
        → 确认事件成立
    即：
        “松手后触发事件”
    该设计能够有效避免：
        长按键重复触发
        机械抖动误触发
        重任务起见重复进入状态机
    在引入状态机后，首先通过“按键触发LED翻转”对系统事件链路进行了再验证
    验证结果表明：
        PH2 按键
            可稳定触发 LED1

        PH3 按键
            可稳定触发 LED2
    该验证表明：
        按键输入
        → 状态机调度
        → 事件触发
        → 上层动作执行
    已能够在当前 Trigger_System 工程中稳定运行。

拍摄与传输解耦
    本阶段中，为了方便验证系统行为，以及问题定位，对系统职责进行了暂时的明确划分：
        PH2：
            仅负责图像采集与缓存刷新
        PH3：
            仅负责发送当前缓存中的图像
    当前系统采用：
        “单槽图像缓存机制”
    即：
        RAM中始终仅保留一张当前有效图像

    同时，本阶段后续出现了“重复拍摄后图像污染问题”
    表现为：
        出现不明原因的色块大面积污染，有时按下传输键后无传输行为发生。
    根据此现象，进行了三组对照实验，一组3~5次重复拍摄：
        统一表现为，上电后拍摄的第一张图片清晰，第二张开始出现色块污染，第三张整张图片污染，直至按下传输键后无法传输图片
    根据三组对照实现推测：
        可能是重复拍摄时，缓存器未被彻底清空，导致RAM中存在大量无效数据。至于按下传输键后无法传输图片，可能时因为Trigger_Event中有len=0时无法触发拍摄行为的防傻瓜机制。而无效垃圾数据无法被有效检测，所以无法触发传输行为。同时也发现，每一次传输的图像数据大小，都比前一次小。
    以上均为推测，其结果需要明天进行进一步验证。
    但此问题也证明了，拍摄与传输的解耦的正确性。若不进行解耦，则很有可能被误导进“摄像头白平衡问题”、“色彩调节问题”、“UART传输问题”

发送握手机制建立
    在完成按键触发机制结构后，昨日建立的MCU ↔ PC握手机制也被迁移至当前事件驱动系统中。
    当前流程为：
        PH3 触发发送
        → MCU 发送 STANDBY
        → PC 返回 READY
        → MCU 开始图像传输
    即：
        每次图像发送前
        都重新执行一次握手流程
    该机制能够确保：
        PC已完成串口初始化，并明确进入接收状态后，MCU才开始图像发送
    同时当前系统中也加入了：
        当jpeg_len == 0时，不允许进入发送流程的保护机制
    
当前阶段验证中
    还发现了PC端通信模块存在：
        单次接收后自动结束的问题
    当前 PC 通信模块流程为：
        打开串口
        → 完成一次完整图像接收
        → 图像保存
        → 程序结束
    因此：
        若希望继续接收下一张图像，
        当前仍需重新运行 Python 通信程序。
    该结构虽适用于初期单次验证链路
    但明显不适用于正在建立的“重复拍摄系统”

AI Agent 辅助开发流程引入
