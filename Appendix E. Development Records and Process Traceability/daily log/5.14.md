日期：5.14

今日任务
    对”重复拍摄导致图像逐步畸变“问题继续展开系统级排查
    重点验证：
        图像畸变问题究竟发生在“图像传输链路”还是“图像采集链路”

通信链路一致性验证：
    首先针对昨日提出的：
        “图像可能在发送/分片/重组过程中发生状态混乱”
        这一怀疑展开验证：
            为避免修改已稳定运行的帧协议，当前未对既有chunk/frame结构进行修改，而是在原有：STANBY/READY握手机制上新增：TX_DONE/DONE_ACK终止握手机制。
        当前完整传输生命周期如下：
            STANDBY → READY → IMAGE TRANSFER → IMAGE COMPLETE → TX_DONE → DONE_ACK → WAIT NEXT STANDBY

图像长度对账实验
    在新增终止握手机制后，进一步加入：
        capture_len
        tx_len
        pc_received_len
        saved_file_size
        四方对账机制。
    
    其中：
        capture_len： MCU端JPEG实际有效长度 
        tx_len： MCU实际发送字节数 
        pc_received_len： PC重组后实际接收长度 
        saved_file_size： PC端保存后的最终文件大小
    连续多轮实验结果表明：
        四项长度完全一致
    
    即：
        capture_len = tx_len = pc_received_len = saved_file_size
    同时：
        图像chunk数量、 last_chunk_size、 image_id、 seq行为也均为发现异常
    因此：
        图像传输链路、chunk分片、PC重组、文件保存基本可以排除为当前问题根因

重新聚焦采集链路
    在通信链路被基本排除后，问题收缩至图像采集链路。
    但昨天已经做了OV2640，DCMI/DMA重置实验，因此怀疑是否是图像抓取逻辑的问题
    事件抓取机制验证
        当前采集逻辑完全复刻开源项目中的机制：
            HAL_Delay() → HAL_DCMI_Stop()固定时间窗口方式结束Snapshot,
            因此开始怀疑当前的重复采集畸变现象是因为的重复拍摄导致的某种“帧错位现象”，多次重复拍摄抓取的JPEG可能并非完整帧，而是在固定时间截取到的一段“合法JPEG”
        故此新增FrameEventDone函数，在拍摄完成后加入超时机制，发现：
            原有的时间抓取逻辑中，FrameDone事件实际上并未触发，JPEG更多属于“时间窗口截取”
        
        故在保留原有时间抓取机制的基础上，新增事件抓取机制-CameraCapture_SnapshotByFrameEvent()实验函数，可使用宏开关随时切换图像抓取机制。
        在事件抓取机制中：
            只有 “done = 1 error = 0 timeout = 0” 时，才允许 “Stop DCMI + 扫描 SOI / EOI + 发送JPEG。”
    连续实验结果表明：
        当前FrameEvent链路已能够正常触发， d
        one状态稳定为1； 
        JPEG边界正常； 
        图像长度合法； 
        图像传输链路完全一致。
    但：
        重复拍摄图像畸变现象依然存在
    因此：
        “固定时间窗口截取”已不是当前问题主音

采集链路全面重置实验
    重新思考“重复拍摄导致图像畸变”这一现象：
        上电后，抓取的第一张照片色彩正常，无畸变现象发生；
        后续随着重复拍摄的次数增加，图像色彩开始畸变，且图像数据大小呈现断崖式下降。
    得出结论：
        一定是某个环节，或整体环节出现了状态残留；
        在昨日，已经做过了：“图像发送后buffer状态清理实验”，“拍摄后OV2640状态重置实验”，”拍摄后DCMI/DMA状态重置实验”
        前面环节已经做过了：“图像数据大小对账实验”，“抓取逻辑变更实验”
        结合以上实验：问题大概率出在图像抓取链路，但尚未知是哪一个环节
    因此决定：
        在图像发送完成后，对图像抓取链路进行一次近乎上电的全面重置
        -Full Acquisition Chain Rebuild实验
        以此判断，问题是否出在图像抓取链路。
        即“先扩大范围，再逐步收缩”
    
    当前流程：
        在图像发送完毕后：
            Stop DCMI 
            → Abort DMA 
            → DCMI/DMA ReInit 
            → I2C/SCCB 重建 
            → PWDN PowerDown/PowerOn 
            → OV2640 Hardware Reset 
            → OV2640 JPEG Init 
            → LightMode Init 
            → 清空 FrameBuffer
    随后进行多轮连续拍摄实验。

    实验结果：
        Full Acquisition Chain Rebuild
        明显改善了：
        “连续拍摄后图像逐步蓝屏/紫屏/染色”
        的累积现象。
    
    当前表现是：
        大部分图片色彩正常；
        图像数据大小在合理范围内随机波动，无明显的断崖式缩小；
        偶有图片出现轻微偏色，但物体边界清晰；
        不再出现此前的：
            第一张照片正常；
            随后图像色彩开始出现畸变；
            持续恶化；
            逐步崩坏至照片中的物体完全不可辨认。
        
当前结论
    当前已经基本可以确认：
        问题并非发生于图像传输链路；
        而属于图像采集链路相关问题。
    
    且：
        单独 OV2640 Reset 
        单独 DCMI/DMA Reset
    均不足以恢复或缓解系统状态

    而：
        接近“重新上电”的完整采集链路重建 
    能够明显打断图像畸变累积。

    当前重点怀疑：
        PWDN PowerDown / PowerOn相关行为
    
    新的关键疑点：
        今日早期还出现了一个异常现象：
            在图像发生畸变后，重新装载程序（热重启），并不能恢复图像状态
        而：
            重新上电后，第一张照片恢复正常
    虽然该现象由于通信热重启容易进入死锁状态，无法稳定复现，
    但结合“Full Rebuild实验 + PWDN控制行为”来看：
        当前“OV2640内部某些状态需通过PWDN接近冷启动方式恢复”
    已成为了新的怀疑方向

下一步计划：
    明日开始，
    将针对：
        Full Acquisition Chain Rebuild
        展开“减法实验”。
    即：
        在保持整体结构稳定的基础上，
        逐步移除：
            DCMI/DMA重建
            I2C/SCCB重建
            PWDN控制
            OV2640 Reset
        等单独恢复项。
    目标：
        确认：
            当前真正起关键恢复作用的模块。
    当前最重点怀疑对象：
        PWDN PowerDown / PowerOn
        以及：
        OV2640内部运行态恢复机制。
