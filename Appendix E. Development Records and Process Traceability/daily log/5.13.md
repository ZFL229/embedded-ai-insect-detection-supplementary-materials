日期：5.13

今日任务目标
    升级PC端通信模块，使其能够完成重复接收图像的动作
    针对当前的Trigger_System工程中出现的“重复拍摄后图像逐步畸变”问题展开系统级排查与验证。

通信模块升级
    此前的通信模块在完成一次图像接收重组后，自动复位
    若需要再次接收图像则需要重新运行，属于初始验证工程

    现在已经完成升级，可重复接收图像

当前异常现象表现为：
    第一次拍摄基本正常；
    后续重复拍摄后，图像逐渐出现整幅后/绿/紫色偏移；
    畸变程度随拍摄次数逐步加深；
    最终可能导致图像几乎完全不可辨认；
    同时伴随JPEG文件长度逐步减小的问题。

问题初步怀疑方向
    最初怀疑方向为：
        JPEG_Buffer中残留上一帧数据
        导致新的JPEG在搜索EOI时提前命中旧EOI
        从而造成JPEG被截断
    因此首先对MCU侧buffer生命周期进行了检查

Buffer残留问题排查
    通过检查camera_capture.c中拍摄逻辑，确认：
        memset(cam_frame_buffer, 0, sizeof(cam_frame_buffer));
    在每次拍摄前均会执行
    同时确认：
        cam_frame_buffer 实际为完整 256KB buffer，
        而非局部清零
    因此：
        “上一帧JPEG Buffer残留导致提前命中旧EOI”这一怀疑证据不足


串口调试与JPEG Marker观察
    由于当前截断Debug模式已经不在适合长期连续观察，因此重新切换为串口调试方式
    通过串口打印观察每次拍摄得到的：
        SOI index
        EOI index
        JPEG Length
        VALID 状态
        DCMI / DMA 状态
    观察结果：
        SOI 始终稳定位于 index = 0；
        JPEG 可正常找到 EOI；
        系统能够持续输出合法 JPEG。
    但同时发现：
        JPEG 长度存在波动。
    在初期怀疑是采集链路内部状态逐渐漂移，
    但事后复盘，发现虽然长度存在波动，但总的来说此波动属于图像采集的正常波动，不至于导致图像畸变。

延时与帧完成机制验证
    当前 Snapshot 逻辑采用：
        HAL_Delay(500);
        HAL_DCMI_Suspend();
        HAL_DCMI_Stop();
    方式结束拍摄

    因此一度怀疑：
        DCMI 尚未真正完成一帧 JPEG，
        系统便提前停止采集，
        导致 JPEG 数据不完整。
    
    随后分别将延时调整为：
        1000ms
        1500ms
        2000ms
    进行验证：
        问题依旧存在。
    因此，简单演示不足并未导致问题的根因。

OV2640 状态残留怀疑与验证
    由于图像畸变呈现出：
        第一次正常；
        后续逐步加深；
        颜色偏移不断累积；
    
    这一“状态漂移”特征，因此开始怀疑：
        OV2640 内部 JPEG / ISP / 输出状态
        在重复拍摄过程中未正确恢复。
    
    随后在每次拍摄完成后加入：
        OV2640 Hardware Reset
        +
        OV2640_Init_1280x960_JPEG()
        +
        LightMode 初始化
    实验结果表明：
        问题依旧存在。
    因此：
        单纯 OV2640 初始化残留
        并不足以解释当前现象。
    
DCMI/DMA状态残留
    随后进一步怀疑：
        DCMI / DMA 采集链路状态机
        在重复拍摄过程中发生状态累积。
    因此设计了三级复位实验：
        轻量复位：
            Stop DCMI
            Abort DMA
            Clear Flag
        中等复位：
            DCMI DeInit / Init
            DMA DeInit / Init
        强复位：
            DCMI Peripheral Force Reset
    并通过宏切换实验等级。

    实验结果：
        三种复位方案均无法解决问题。
    因此当前可以初步认为：
        问题并非简单 DCMI/DMA 状态残留。

重新审视“图像传输链路”
    在晚间复牌过程中，重新检查此前串口调试日志时，发现了一个此前被忽略的重要现象：
        串口打印的JPEG长度：
            约 29KB ~ 40KB 波动
        虽然存在差异，但整体差异呈现波动状，比如第一张36kb，第二张34kb，第三张35kb
        但在PC端实际接收到的图像中：
            第一张约 60~65KB
            第二张约 45KB
            第三张约 35KB
        呈现出：
            “拦腰截断式下降”
    这张变化已经明显超出正常JPEG压缩差异

    同时：
        图像畸变程度与数据长度下降同步增强；
        后续图像逐渐出现整体偏色与严重失真
    由此产生了新的关键怀疑：
        极大可能是某个地方出现了状态残留，但
        这个状态残留未必发生在“图像采集截断”
        也可能发生在“图像传输/分片重组阶段”
    即：
        MCU可能已采集到完整 JPEG，
        但在发送、分片、重组或保存过程中，
        出现了状态残留、提前结束、长度错误、
        chunk 状态错乱等问题。
    但是同时保留对图像采集链路的怀疑。

当前系统状态
    当前系统已确认：
        连续拍摄问题并非简单 buffer 污染；
        延长采集时间无法解决问题；
        OV2640 单独重初始化无法解决问题；
        DCMI/DMA 多级复位仍无法彻底恢复；
        图像异常具有明显“链路状态累积”特征；
        图像传输链路已成为新的重点怀疑对象。
    下一步计划
        1.对“采集长度 / MCU发送长度 / PC保存长度”
            进行三方对账验证。
            重点检查：
            - chunk 状态是否正确复位；
            - total_chunks 是否正确更新；
            - image_id / seq 是否发生错乱；
            - PC 是否提前保存；
            - 发送长度与保存长度是否一致。
        
        2.Full Acquisition Chain Rebuild 实验
            即：
            每次图像发送完成后，
            重新执行接近“上电初始化”的采集链路重建。
            重建范围包括：
            DMA
            DCMI
            I2C/SCCB
            OV2640 Hardware Reset
            OV2640 JPEG Init
            但保留：
            UART
            GPIO
            按键状态机
            PC 通信模块
            实验目标：
            验证当前问题是否属于“采集链路运行态漂移”。
            若问题消失：
            则证明某个采集链路状态未正确恢复。
