    明确系统工作模型（关键认知突破）
        建立完整数据与控制分层：
            控制链路：
                STM32 → SCCB(I2C) → 摄像头寄存器

            数据链路：
                摄像头 → DVP(并口) → DCMI → DMA → 内存
        明确：
            DCMI只负责“收数据”
            SCCB才是“控制摄像头”的核心

    发现关键问题：PWDN不在MCU上
        通过开发板原理图确认：
            摄像头PWDN并未直接连接在STM32 GPIO
            而是连接至PCF8574 I2C IO扩展芯片的P2引脚
        即：
            STM32 → I2C → PCF8574 → P2 → 摄像头 PWDN
        结论：
            MCU当前无法直接控制摄像头电源状态
            摄像头很可能一直处于Power Down状态
    
    问题根因定位：
        解释了此前现象：
            无法读取摄像头ID
            无法进行后续DCMI数据采集
        根本原因：
            摄像头未被唤醒（PWDN未拉低）

    架构调整
        将当前最小验证目标重新定义为：
            1. 控制 PCF8574
            2. 拉低 PWDN（开启摄像头）
            3. 通过 SCCB 读取 OV2640 ID
    
        明确当前阶段优先级：

关键理解与收获：
    摄像头系统不是“一个外设”，而是一个子系统
        包含：
            控制总线(SCCB)
            数据总线(DCMI)
            电源控制(PWDN/RESET)
            时钟系统(内部/外部)
    
    DCMI不等于摄像头初始化
        DCMI只是“采集器”，不是“控制器”
    
    调试顺序必须分层:
        当前正确路径为:
            摄像头供电 → SCCB通信 → 读ID → 再考虑图像采集
        而不是一开始就做:
            DCMI + DMA + 图像

    找到系统真正的开关
        PWDN 是摄像头是否工作的关键控制点
