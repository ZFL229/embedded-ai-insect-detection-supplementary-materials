F407板载ST-LINK作为外部调试器验证
    初始目标
        由于当前H7开发板缺乏稳定的Debug能力，决定利用STMF407G-DISC1开发板的板载ST-LINK模块，作为外部调试器接入H7
    
    硬件连接与问题排查过程
        连接方式：
            SWDIO → SWDIO 
            SWCLK → SWCLK 
            GND → GND
        在调试过程中，出现以下典型问题：
            ST-LINK能识别，但无法获取Target Core ID
            CubeProgrammer显示DEV_TARGET_CMD_ERR
            识别对象始终为F407而非H7
            有时可以连接成功，有时不行
    
    关键问题定位
        最终定位问题为：
            SWD六针接口上的GND存在接触或连接问题
        表现为：
            有时连接成功有时不行
            使用该GND时连接不稳定或失败
            更换至ST-LINK四针接口上的GND后立即稳定
    
    最终稳定方案：
        SWDIO → CN2 (SWD) 
        SWCLK → CN2 (SWD) 
        GND → ST-LINK 四针接口 GND
    
    成功实现：
        STM32CubeProgrammer正确识别H7(Device ID:0x450)
        CubeIDE可正常烧录程序
        Debug模式可进入
    
    工程意义：
        该步骤完成后
            H7开发进入可调式状态
            不再依赖串口打印
            为后续摄像头信号调试提供了基础条件
    
Debug基础能力学习与验证
    调试能力打通后，转而学习Debug使用方法，基于按键检测工程进行验证
    
    Debug操作掌握
        已掌握：
            断点（Breakpoint）
            单步执行（Step Into / Step Over / Step Return）
            程序执行指针（PC）
            Resume 控制程序运行
    能够跟踪程序从main.c → bsp_key.c → HAL_GPIO_ReadPin的执行路径

    变量观察方法
        掌握Expressions窗口的正确使用：
            仅添加变量名(避免表达式匹配错误)
            使用volatile防止优化影响Debug结果
        观察变量：
            g_key_state 
            now 
            g_key_tick
    
    GPIO电平变化验证
        通过Debug模式观察完成以下链路验证：
            按键输入 → GPIO电平变化 → HAL读取 → 状态机更新 → event产生 → 主循环响应
        验证结果：
            GPIO电平随按键变化
            状态机按预期实行(IDLE → PRESS → RELEASE)
            event正确触发
            LED控制逻辑正常执行
    
    Debug方法论总结
        形成分层调试思路：
            0️⃣ 程序执行路径（断点） 
            1️⃣ 变量层（HAL读取结果） 
            2️⃣ 寄存器层（GPIO IDR） 
            3️⃣ 硬件层（真实信号）
    并形成关键认知：
        Debug中的变量本质式"观测系统状态的探针"，而非普通程序数据
